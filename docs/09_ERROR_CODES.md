# ERROR CODES (Phase 0.4)

تمامی توابع صادر شده از لایه C، مقادیر `int` برمی‌گردانند که نمایانگر وضعیت اجرای دستور است. عدد `0` به معنای موفقیت و اعداد منفی نمایانگر خطا هستند.

## لیست کدهای خطا (Error Codes)

| شناسه ثابت (Macro در C) | مقدار `int` | دسته‌بندی استراتژیک (Error Strategy) | توضیحات خطا |
| :--- | :--- | :--- | :--- |
| **`RE_OK`** | `0` | - | عملیات با موفقیت انجام شد. |
| **`RE_ERR_VALIDATION`** | `-1` | **User Error** | داده ورودی (JSON یا استرینگ) نامعتبر است (مثلاً کد ملی نامعتبر، خالی بودن فیلد اجباری، یا قیمت منفی). |
| **`RE_ERR_NOT_FOUND`** | `-2` | **User Error** | موجودیت درخواست شده (مانند آیدی ملک) در دیتابیس یافت نشد. |
| **`RE_ERR_DUPLICATE`** | `-3` | **User Error** | اطلاعات وارد شده تکراری است (مثلاً ثبت کاربر با نام کاربری یا کد ملی که از قبل وجود دارد). |
| **`RE_ERR_AUTH`** | `-4` | **User Error** | احراز هویت ناموفق بود (نام کاربری یا رمز عبور اشتباه است، یا توکن سشن منقضی شده). |
| **`RE_ERR_LOCKED`** | `-5` | **User Error** | حساب کاربری به دلیل تلاش‌های ناموفق متعدد قفل شده است. |
| **`RE_ERR_FORBIDDEN`** | `-6` | **User Error** | کاربر لاگین است، اما سطح دسترسی (Role) لازم برای انجام این عملیات را ندارد (مثلاً یک Agent که سعی در حذف کاربر دیگر دارد). |
| **`RE_ERR_DB`** | `-7` | **Fatal / Recoverable** | خطای پایگاه داده هنگام اجرای کوئری (سینتکس، فیلد ناموجود). |
| **`RE_ERR_BUSY`** | `-10`| **Retryable Error** | پایگاه داده به دلیل عملیات سنگین شخص دیگر موقتاً قفل است. (کلاینت باید با تاخیر چند میلی‌ثانیه‌ای مجدد تلاش کند). |
| **`RE_ERR_CORRUPT`** | `-11`| **Fatal Error** | فایل دیتابیس SQLite آسیب دیده و ساختار آن مخدوش است. برنامه باید بسته شود. |
| **`RE_ERR_MEM`** | `-98`| **Fatal Error** | کمبود حافظه (Memory Allocation Failed) در لایه C هنگام اجرای عملیات. |
| **`RE_ERR_INTERNAL`** | `-99`| **Fatal Error** | خطای ناشناخته و داخلی سیستم (مثلاً باگ در کتابخانه‌های وابسته یا عدم دسترسی به دیسک برای خواندن فایل). |
| **`RE_ERR_NOT_IMPLEMENTED`** | `-100`| **Fatal Error** | تابع درخواستی در معماری API اکسپورت شده است اما منطق (Implementation) آن هنوز در لایه Repository تکمیل نشده است. |

## مکانیزم تبدیل در پایتون (Exception Mapping)
لایه `re_bridge` در پایتون، پس از فراخوانی هر تابع DLL خروجی `int` آن را بررسی می‌کند.
در صورتی که خروجی عددی غیر از `0` باشد، تابع کمکی `raise_for_error(status_code)` اجرا شده و اکسپشن پایتونی متناظر (مانند `ValidationException(Exception)`) را Raise می‌کند.
این امر باعث می‌شود برنامه‌نویس UI هرگز درگیر کدهای عددی منفی نشود و صرفاً بلوک‌های `try...except` پایتونی بنویسد.
