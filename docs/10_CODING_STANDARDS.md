# CODING STANDARDS (Phase 0.5)

این سند استانداردهای کدنویسی پروژه را به تفکیک زبان‌ها و تکنولوژی‌های استفاده شده تعیین می‌کند تا یکپارچگی کد در بالاترین سطح صنعتی حفظ شود.

## 1. زبان C (لایه Core و DLL)
- **استاندارد زبان:** C17.
- **سطح هشدارها (Warnings):** استفاده از `/W4` در MSVC و `-Wall -Wextra -Werror` در GCC/Clang (هیچ Warningای در کد قابل قبول نیست).
- **Const Correctness:** تمامی پارامترهای ورودی و پوینترهایی که نباید تغییر کنند **باید** `const` باشند (مخصوصاً `const char*`).
- **قانون مالکیت حافظه (Ownership Rule):** توابعی که رشته یا ساختاری تخصیص داده شده (malloc) برمی‌گردانند، مالکیت آن را به فراخواننده (Caller) منتقل می‌کنند. پایتون موظف به فراخوانی `re_free_string` است.
- **Error Handling & Cleanup:** مدیریت منابع داخلی C با استفاده از الگوی `goto cleanup` جهت جلوگیری از نشت حافظه یا بازماندن فایل‌ها انجام می‌شود.
- **Thread Safety:** توابع بیزینسی Stateless هستند. در صورت استفاده از State، باید از Mutex استفاده شود.
- **ترتیب Includeها (Include Order):**
  1. هدر فایل متناظر (`#include "my_module.h"`)
  2. هدرهای استاندارد C (`<stdio.h>`)
  3. هدرهای وابستگی‌ها (`"sqlite3.h"`)
  4. هدرهای داخلی پروژه
- **نام‌گذاری (Naming Convention):** تمامی متغیرها، توابع و فایل‌ها به صورت `snake_case`. توابع عمومی (Exported) باید با `re_` شروع شوند.
- **مستندسازی (Documentation):** استفاده از استاندارد Doxygen برای کامنت‌گذاری توابع در هدر فایل‌ها (`/** ... */`).

## 2. زبان Python (لایه Bridge و UI)
- **استاندارد فرمت:** رعایت کامل PEP8 با اعمال اتوماتیک توسط ابزار **Black** و مرتب‌سازی ایمپورت‌ها با **isort**.
- **Linting:** استفاده از **Ruff** برای آنالیز کد.
- **Type Hinting:** استفاده از Type Hints برای تمامی ورودی‌ها و خروجی‌های توابع **اجباری** است.
- **مدل‌های داده:** استفاده از `dataclasses` (یا `pydantic`) جهت ساخت مدل‌های خوش‌تعریف برای نگاشت داده‌های JSON.
- **سیاست لاگینگ (Logging Policy):** استفاده از ماژول `logging` استاندارد پایتون. استفاده از `print()` ممنوع است.
- **سیاست استثناها (Exception Policy):** لایه Bridge باید کدهای خطای منفی C را دریافت و منحصراً به Custom Exceptionهای پایتون (مثل `AuthException`, `ValidationException`) تبدیل و Raise کند. UI فقط باید این Exceptionها را Catch کند.

## 3. پایگاه داده (SQL / SQLite)
- **نام‌گذاری جداول و ستون‌ها:** `snake_case`.
- **مایگریشن‌ها:** فرمت فایل‌های مایگریشن به صورت `<version>_<description>.sql` (مثل `0001_initial.sql`).
- **نام‌گذاری ایندکس‌ها:** `idx_<table>_<column>` (مثال: `idx_users_role`).
- **نام‌گذاری تریگرها:** `trg_<table>_<action>` (مثال: `trg_properties_audit`).
- **نام‌گذاری ویوها:** `vw_<name>` (مثال: `vw_active_properties`).
- **قانون تراکنش (Transaction Rule):** هر عملیات ترکیبی دیتابیس (Insert/Update چندگانه) باید اکیداً در یک بلاک `BEGIN TRANSACTION` تا `COMMIT/ROLLBACK` محصور باشد.

## 4. تبادل داده (JSON DTOs)
- **نام‌گذاری کلیدها:** طبق درخواست، تمامی کلیدهای داخل JSON باید به فرمت **`camelCase`** باشند (مثال: `userId`, `municipalDistrict`). لایه C و Python باید به درستی `camelCase` را به `snake_case` داخل کد متصل کنند.
- **فیلد نسخه (Version Field):** هر پاسخ JSON باید دارای فیلد `apiVersion` باشد.
- **سیاست Nullable:** فیلدهایی که ممکن است خالی باشند باید در JSON به صراحت با مقدار `null` ارسال شوند (حذف فیلد از بدنه JSON ممنوع است تا ساختار DTO ثابت بماند).
