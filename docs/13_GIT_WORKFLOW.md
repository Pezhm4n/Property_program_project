# GIT WORKFLOW (Phase 0.5)

این سند فرآیند استاندارد توسعه نرم‌افزار، مدیریت Branchها و چرخه یکپارچه‌سازی مداوم (CI/CD) را مشخص می‌کند.

## 1. مدل توسعه (Trunk Based Development / GitHub Flow)
پروژه از سیستم ساده‌سازی شده GitHub Flow استفاده می‌کند تا سرعت یکپارچگی کد بالا بماند:
- شاخه `main`: منبع اصلی و پایدار پروژه (حاوی کدهای تست شده و آماده توزیع).
- تمامی توسعه‌ها روی شاخه‌های فرعی (Feature Branches) انجام می‌شوند.
- ادغام با `main` فقط از طریق **Pull Request (PR)** و تأیید Code Review انجام می‌پذیرد.

## 2. قرارداد نام‌گذاری شاخه‌ها (Branch Naming)
- قابلیت‌های جدید: `feature/<task-name>` (مثال: `feature/add-audit-log`)
- رفع باگ‌ها: `bugfix/<issue-name>` (مثال: `bugfix/fix-login-crash`)
- بهبودهای پرفورمنس یا زیرساخت: `chore/<description>`

## 3. قرارداد پیام کامیت (Conventional Commits)
پیام تمامی کامیت‌ها باید ساختار مشخص زیر را داشته باشند تا تولید Changelog خودکار امکان‌پذیر شود:
- `feat:` ویژگی‌های جدید
- `fix:` رفع باگ‌ها
- `docs:` تغییرات در مستندات پروژه
- `style:` تغییرات فرمتینگ کد (بدون تغییر لاجیک)
- `refactor:` بازنویسی کدهای موجود جهت بهبود معماری
- `test:` اضافه شدن یا اصلاح تست‌ها
- `chore:` تنظیمات سیستم بیلد یا CI/CD

## 4. قالب Pull Request و چک‌لیست Code Review
هر Pull Request باید چک‌لیست زیر را پاس کند:
- [ ] تست‌های مرتبط نوشته یا آپدیت شده‌اند.
- [ ] کد استانداردهای Linting پایتون (Ruff/Black) را پاس می‌کند.
- [ ] در صورت تغییر API لایه C، مستندات (از جمله `07_API_SPECIFICATION.md`) به‌روز شده‌اند.
- [ ] بیلد روی تمام محیط‌های تعیین شده سبز است (CI Passed).
- [ ] تاییدیه حداقل یک توسعه‌دهنده ارشد (Code Review Approved).

## 5. چرخه‌ی CI/CD (طراحی پایپ‌لاین)
حتی پیش از پیاده‌سازی کامل اسکریپت‌ها، معماری پایپ‌لاین CI به این شکل طراحی شده است:
1. `Git Push / Pull Request` ➔ تریگر شدن GitHub Actions
2. `Build C` ➔ اجرای CMake و کامپایل DLL روی ماشین مجازی لینوکس/ویندوز.
3. `Run Unit Tests` ➔ اجرای `ctest` و بررسی Memory Leak با `Valgrind` (فقط لینوکس).
4. `Python Tests` ➔ نصب لایه Bridge و اجرای `pytest`.
5. `Linting` ➔ اجرای Ruff/Black روی پایتون و نمایش اخطارهای کامپایلر C.
6. `Release Artifact` ➔ (فقط روی تگ زدن) باندل کردن اتوماتیک فایل `.exe` توسط PyInstaller و آپلود به GitHub Releases.

## 6. انتشار و نسخه‌گذاری (Release Tagging)
هر زمان که یک نسخه قابل توزیع (پایدار) روی `main` شکل بگیرد، یک تگ SemVer روی گیت قرار می‌گیرد (مانند `v1.0.0`) که به صورت خودکار فرآیند ساخت Installer و آپلود در GitHub Release را فعال می‌کند.
