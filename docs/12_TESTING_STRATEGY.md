# TESTING STRATEGY (Phase 0.5)

استراتژی تست سیستم برای تضمین پایداری معماری و قابلیت اطمینان، بر اساس پوشش گسترده تمامی لایه‌ها تدوین شده است.

## 1. پوشش تست (Coverage Target)
پوشش خطوط کد (Code Coverage) برای منطق کسب‌وکار (C Core) و لایه Bridge پایتون **حداقل 80%** تعیین شده است. هیچ کدی بدون پاس شدن تمامی تست‌ها به شاخه `main` ادغام نخواهد شد.

## 2. دسته‌بندی تست‌ها

### 2.1 Unit Tests (تست‌های واحد)
- **C Core:** با استفاده از فریم‌ورک (Unity یا Criterion). تست مستقل ماژول‌های اعتبارسنجی (مانند صحت کد ملی)، پارسر JSON و محاسبات هش (بدون دیتابیس واقعی).
- **Python Bridge:** با استفاده از `pytest`. تست ماک شده کلاس‌های Bridge و تطابق نگاشت ارور کدها به اکسپشن‌ها.

### 2.2 Integration Tests (تست‌های یکپارچگی)
تست روی دیتابیس SQLite واقعی اما در یک فایل در حافظه موقت (Temporary DB) یا `memory:`.
هدف این تست اطمینان از صحت کارکرد پیوسته `Service -> Repository -> SQLite` و بازگشت صحیح DTOهای JSON است.

### 2.3 API Boundary Tests (تست‌های مرز لایه‌ها)
تست‌های متمرکز بر روی `ctypes` که مطمئن شوند ABI نشکسته است، تایپ‌ها به درستی پاس داده می‌شوند و نشت حافظه اتفاق نمی‌افتد.

### 2.4 GUI Tests (تست‌های رابط کاربری)
با استفاده از کتابخانه `pytest-qt` برای تست کلیک دکمه‌ها، اعتبار‌سنجی فرم‌ها پیش از ارسال به لایه C و تغییر ویوهای برنامه. این تست‌ها در محیط Headless لینوکس (توسط Xvfb) نیز اجرا می‌شوند.

### 2.5 Regression Tests (تست پسرفت)
با افزوده شدن هر ویژگی یا گزارش باگ، یک تست جدید نوشته می‌شود تا مطمئن شویم باگ‌های قبلی دوباره ظاهر نمی‌شوند.

### 2.6 Performance Tests (تست عملکرد)
اجرای کوئری‌های پیچیده روی دیتابیسی با 1,000,000 رکورد ساختگی (Mock Data) تا اطمینان حاصل شود که زمان پاسخ‌دهی (Response Time) در تمام شرایط **زیر 200 میلی‌ثانیه** باقی می‌ماند.

### 2.7 Security Tests (تست امنیت)
- **SQL Injection Prevention:** ارسال عمدی کاراکترهای مخرب و بدافزار از پایتون به لایه C جهت اطمینان از عملکرد صد درصدی Prepared Statements.
- **Auth Flow Testing:** تست قفل شدن حساب بعد از 5 بار تلاش ناموفق و مدیریت صحیح انقضای Session Token.
